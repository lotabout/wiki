6 Streams

8 Quene's Problem.

Using backtrack is hard to understand. That's because it takes time into consideration.

Using stream is a lot more easier.

One problem of stream is that we will have to construct all the stream items before using it. In some cases they are not even needed.

Thus we need lazy evaluation. That is so-called "On Demand Data Structure".

Actually, if we delay all the arguments when applying a function, we'll natually get the power of stream processing.

One problem this introduce is that lazy evaluation is gonna miss up with assignment. This is why functional programming language like "Haskell" don't have assignment at all.

The essence behind this is time information. With pure functional programming, there is no way to actually get the time information. And this affects our view of the word in programming.

7 Metacircular Evaluation

Y Combinator 

    Y = (λ (f)
           (λ (x x) (f (x x)))
           (λ (x x) (f (x x))))
    (Y F) => (F (F (F ... )))

That means (Y F) is a way to find a fix point of function F. Thus if we iterate the eval and apply, we'll finally get what Lisp is.

One more thing to note is the convergence of function F. Normally it's verty hard to decide whether or not F is convergent at the very beginning.

Lexical Binding VS Dynamic Binding.

Lexical Binding is when construct 'lambda' expressions, we also capture its environments, while bynamic binding evaluates a 'lambda' according its running environment.

Dynamic binding has a very serious problem: modularity disaster, which is that when we write a module, the names we use in it matters outside the module.

How to get dynamic binding feature in a lexical environment? => Wrap it with a 'lambda', who captures the dynamic variables.
